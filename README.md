# ⚙️ TaskForge

Most "task manager" projects are a to-do list with extra steps.

> This one's actually a data-structures showcase in disguise: the scheduler underneath runs a **max-heap** for priority ordering, a **hash map** with separate chaining for O(1) lookup, and **twin stacks** for undo/redo — all in C++, compiled to WebAssembly, running in your browser tab.

The JavaScript draws the screen; every decision is made in compiled C++.

🔗 **Live app:** [seharish-fatima.github.io/TaskForge](https://seharish-fatima.github.io/TaskForge/) — real C++ running via WebAssembly, no install.

---

## 🚀 Overview

A to-do app on the surface. A DSA course's greatest hits underneath, actually load-bearing instead of theoretical.

## 🧠 Why It Exists

Heaps, hash maps, and stacks usually live in a textbook or a leetcode grind, proven correct on paper and never touched again. This puts them where they can't fake it — behind a UI real tasks pass through, compiled to WASM, with a native test harness holding them to their invariants before the browser ever sees them.

---

## 🎮 What It Actually Does (no cap)

**Priority scheduling via a real heap** — every task has a 1–10 priority, and the "next up" banner is literally the top of a max-heap maintained in C++. Complete the top task and the heap re-surfaces the next highest automatically.

**Undo / redo that actually works** — backed by two stacks of task snapshots in the engine, not a JavaScript hack. Undo a completion, a deletion, or an edit; redo re-applies it.

**O(1) task lookup** — a hash map with separate chaining finds any task by id without scanning the list. Overkill for a to-do app, which is the point: it's the DSA that matters.

**Persistence** — tasks save to your browser's localStorage, and on reload the C++ engine **rebuilds its heap and hash map from the saved rows**. Nothing leaves your machine.

**Runs in the browser** — the whole engine is `emcc`-compiled to WASM and bound to JavaScript with Embind. Click the link; you're running compiled C++.

## 🛠 Project Structure

```
TaskForge/
├── src/
│   ├── engine.h          # heap, hash map, undo stack, Task — pure data structures
│   └── scheduler.cpp     # Scheduler facade, JSON API, Embind bindings + native test harness
├── web/
│   ├── index.html        # UI shell
│   ├── app.js             # JS glue — calls into the WASM Scheduler, renders
│   └── scheduler.js/.wasm # emcc output (generated by build.sh)
└── build.sh               # one-command Emscripten compile
```

`src/` doesn't know a browser exists. The same `scheduler.cpp` compiles two ways: with `emcc` it becomes the WASM module the web UI drives, and with plain `g++` it builds a **native test harness** that asserts the engine's behaviour — heap ordering, undo/redo across complete/remove/modify, JSON export. The logic is verified by the g++ build before `emcc` ever retargets it to WASM.

## ▶️ Build & Run

You need [Emscripten](https://emscripten.org/docs/getting_started/downloads.html) once:

```bash
git clone https://github.com/emscripten-core/emsdk && cd emsdk
./emsdk install latest && ./emsdk activate latest && source ./emsdk_env.sh
cd ..
```

Then compile and serve:

```bash
./build.sh                 # produces web/scheduler.js + web/scheduler.wasm
cd web && python3 -m http.server 8000
```

Open `http://localhost:8000`. (WASM won't load from a `file://` path — it needs to be served, hence the tiny http.server.)

Verify the engine logic natively without Emscripten:

```bash
g++ -std=c++17 src/scheduler.cpp -o test && ./test
```

Or skip all of that — it's already live: **[seharish-fatima.github.io/TaskForge](https://seharish-fatima.github.io/TaskForge/)**
